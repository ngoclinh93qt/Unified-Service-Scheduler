The Unified Service Scheduler

A backend that confirms an appointment only when a service bay and a qualified technician are available for the full service time.

1. Executive summary

Design goal. Confirm an appointment only when the vehicle, a service bay and a qualified technician are available for the full service time.
Design approach. The backend checks availability, selects the resources and creates the appointment in one PostgreSQL transaction. Database rules prevent overlapping bookings, including requests made at the same time.

2. Requirements, assumptions and scope

This section defines the requirements, assumptions and design decisions used in this solution.

GIVEN stated requirement   INFERRED needed for correctness   ASSUMED design assumption   DECIDED scope decision

ID Requirement Type
R1 Accept a request for a specific vehicle, service type, dealership and desired start time. GIVEN
R2 Before confirmation, find one available service bay and one qualified technician for the entire service duration. GIVEN
R3 Persist the confirmed appointment with customer, vehicle, service bay and technician associations. GIVEN
R4 Concurrent requests cannot double-book the vehicle, bay or technician. INFERRED
R5 Service duration is configured by service type; the server computes the end time. ASSUMED
R6 The current design does not automatically handle duplicate booking requests. DECIDED

Explicit assumptions

  • One appointment requires exactly one bay and one qualified technician.
  • Appointments are stored in UTC and use [start, end), so one appointment can start exactly when another ends.
  • The server calculates the end time by adding the service duration to the requested start time. Business hours can be added later using the dealership's timezone.
  • Bays are interchangeable in the first release; technician qualification depends on service type.
  • Dealerships, vehicles, service types, bays, technicians and skills already exist before a booking is created.

Deliberately out of scope

Authentication and authorization are outside the current scope. In production, customer and dealership access would come from the signed-in user, and access to customer and vehicle data would be audited. Notifications, cancellation, rescheduling and multi-region writes are also out of scope.

3. Architecture and component responsibilities

Component Responsibility Boundary
HTTP adapter Validates HTTP requests, adds request IDs, returns consistent responses and provides the OpenAPI contract. Does not select resources.
Application Checks the booking time, coordinates the booking flow, records the result and calls the booking port. Does not depend on Prisma types.
Domain Calculates service time ranges and checks whether they overlap. No framework or I/O imports.
Persistence adapter Runs the PostgreSQL transaction, finds suitable resources, locks rows and converts database errors. PostgreSQL-specific behavior stays here.
PostgreSQL Stores appointments and rejects overlapping bookings. Confirmation exists only after commit.

Technology choices

NestJS
Organizes the backend into clear modules and provides validation and OpenAPI support.
PostgreSQL
Transactions, row locks and exclusion constraints directly prevent double booking.
Prisma
Provides type-safe database access and migrations. Raw SQL is used only for PostgreSQL locking features that Prisma does not expose.
Testcontainers
Runs automated concurrency tests against real PostgreSQL instead of an in-memory database.

4. Booking flow and concurrent requests

Transaction approach. Each booking runs in one PostgreSQL READ COMMITTED transaction. The transaction keeps reference data unchanged, then locks suitable bays and technicians in a fixed order. After any wait, it checks availability again before creating the appointment.
Known trade-off. Locking all suitable resources means booking requests for the same dealership may wait for each other. This is acceptable for the assumed load. The system can lock fewer rows later if waiting becomes frequent.
  1. Start one transaction, validate the referenced data, load the service duration and calculate the UTC time range.
  2. Lock suitable bays, qualified technicians and their qualification records in a fixed order.
  3. Check the start time and availability of the vehicle, bays and technicians again after acquiring the locks.
  4. Select the first available pair, create the appointment and commit. Return success only after the commit; otherwise roll back and return the appropriate error.

How double booking is prevented

If availability is checked before the transaction, two requests may select the same resources. The availability check and appointment creation therefore happen in the same transaction.

Control Purpose Guarantee
Request and time validation Reject invalid booking requests. Valid input.
Locked reference data Prevent referenced data from changing during booking. Consistent customer, vehicle and service data.
Resource locks and second availability check Coordinate booking requests made at the same time. Uses current availability before selection.
PostgreSQL exclusion constraints Reject overlapping vehicle, bay or technician use. Final database protection against double booking.

After waiting for a lock, a request can select another available pair. Database constraints remain the final protection if the normal booking flow is bypassed. A fixed lock order reduces deadlock risk. A temporary database or lock problem returns 503, while an overlapping booking returns 409.

Resource selection

Find suitable bays and qualified technicians, remove those already booked for the requested time, then choose the first free pair by ID. This is simple and predictable. Fair or load-balanced scheduling can be added later.

Rejected alternatives

Alternative Why it was not selected
SERIALIZABLE Correct, but PostgreSQL may cancel transactions that then need retries. Explicit resource locks are simpler for the assumed load.
SKIP LOCKED May skip a resource held by a transaction that later rolls back, causing a false unavailable result.
Redis/distributed lock Adds another locking system even though PostgreSQL already stores the booking data.
Microservices / CQRS / Kafka The booking flow needs one transaction, so splitting it into multiple services adds unnecessary complexity.

5. Data model and database rules

The ERD shows the main data needed for the booking flow. Audit columns and secondary indexes are omitted for clarity.

Database rules

  • end_time > start_time.
  • Confirmed intervals cannot overlap for the same vehicle_id.
  • Confirmed intervals cannot overlap for the same service_bay_id.
  • Confirmed intervals cannot overlap for the same technician_id.
  • The assigned technician must be qualified and all assigned resources must belong to the requested dealership. These rules are checked inside the transaction.

The status field allows a future cancellation to release resources without deleting appointment history. Only CONFIRMED appointments reserve capacity.

6. API and error handling

The API creates appointments, reads confirmed appointments and reports service health.

Availability check. The booking request checks current availability before confirmation. A separate GET /availability endpoint can later provide a preview for the UI, but the final check remains part of POST /appointments.

GET /api/v1/appointments/{id} Read the persistent confirmed record. 200 / 400 / 404

The current API accepts customerId in the body and checks that the customer owns the vehicle. In production, the customer identity should come from the signed-in user. startTime must be in the future.

Errors

All failures use application/problem+json (RFC 9457) with a consistent error type and request ID.

Status Meaning Example
400 The request is invalid. Invalid timestamp, unknown field, non-UUID id, or a startTime in the past.
404 Required data cannot be found. Unknown vehicle or service type.
409 · RESOURCES_UNAVAILABLE The vehicle is already booked for the interval, or no complete bay and qualified technician pair remains. Client selects another start time.
409 · REFERENCE_CONFLICT The request contains conflicting data, such as a vehicle belonging to another customer. Client corrects the request references.
503 · TRANSIENT_FAILURE A temporary database problem, such as a deadlock or lock timeout. Client may retry after the Retry-After hint. Automatic server retry can be added later.
500 Unexpected failure; internals are not exposed. Request ID returned for support.

Future: safe request retries

A future version can accept an Idempotency-Key. Repeating the same request with the same key returns the original result, while using the key with different data is rejected.

7. Reliability, operations and verification

Verification strategy

Test Expected result Environment
Interval construction A valid start time and duration produce the correct UTC end time; invalid input is rejected. Pure unit tests
Resource selection Only a valid bay and qualified technician can be selected. Application + PostgreSQL
Database constraint Direct vehicle, bay and technician conflicts are rejected even when data is written outside the normal API. Real PostgreSQL
Concurrent race When two requests compete for one resource pair, exactly one succeeds and one appointment is stored. Testcontainers + PostgreSQL
Concurrent bookings with two pairs Two different vehicles can use separate resource pairs at the same time, while the same vehicle cannot be booked twice. Testcontainers + PostgreSQL

Runtime and deployment

Docker Compose starts the API and PostgreSQL, applies database migrations and loads demo data. Appointments remain after a restart. In production, migrations and demo data setup would run separately from application startup.

Observability

The current version writes structured JSON logs with request IDs. Metrics and tracing are planned for a later version.

  • Logs: include the request ID, booking result, dealership, service type and processing time. Authorization and cookie values are removed.
  • Metrics (planned): booking results, booking time, conflicts, database lock waits and connection-pool usage.
  • Tracing (planned): follow a booking from the HTTP request through validation, resource selection and database commit.
  • Alerts (planned): notify the team about repeated server errors, long lock waits or a nearly full connection pool. Alert thresholds will be set after load testing.

Capacity and future scaling

The design assumes a small dealership with modest booking traffic. If load tests show many requests waiting or slow bookings, the system can check resources in one query and lock fewer rows. Database rules will continue to prevent overlapping bookings.

8. AI collaboration during design

I used AI to explore design options and identify possible problems. I provided the context, reviewed the suggestions and made the final decisions.

1 · Requirements
I used AI to identify missing details and possible assumptions. I reviewed the options and chose the requirements and assumptions used in this design (section 2).
2 · Architecture
I described the user flow, data flow and expected components. AI suggested different structures, and I chose a modular monolith because it keeps the booking transaction simple while preserving clear module boundaries.
3 · Failure modes
AI helped me compare SERIALIZABLE, SKIP LOCKED, database constraints and distributed locks. Based on the assumed small scale, I chose to lock all suitable resources. This makes concurrent bookings wait, but keeps the design simple. I rejected a separate “check then insert” flow because two requests could select the same resources.
4 · API & data
I chose REST, RFC 9457 error responses and the main data relationships. AI helped draft the API and data model, which I reviewed and corrected.
5 · Observability
I chose structured logs with request IDs for the current design. Metrics and tracing remain future design work.
6 · This document
AI helped draft each section from my direction. I reviewed the structure, technical decisions and final wording.
Data boundary. No secrets, production credentials or customer data were shared with AI tools. I reviewed AI output before including it in the design.
