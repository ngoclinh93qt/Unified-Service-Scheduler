The Unified Service Scheduler

A correctness-first backend that confirms an appointment only after one service bay and one qualified technician have been reserved for the complete service interval.

Central invariant. Two confirmed appointments must never reserve the same vehicle, service bay or technician for overlapping time intervals.
Implementation status (this milestone). The implementation provides the POST/GET booking slice, application and database-time validation, the transactional allocation path (share-locked reference rows plus exclusively locked candidate rows), PostgreSQL GiST exclusion constraints for vehicle, bay and technician, precise transient-failure classification to 503, credential redaction, structured booking outcome logging and graceful Prisma disconnect. Idempotent replay, internal retry/backoff, principal-derived identity and metrics/tracing are explicitly deferred.

2. Requirements, provenance and scope

The brief is intentionally ambiguous. The labels below distinguish official requirements from engineering deductions and submission choices.

GIVEN official brief   INFERRED required for correctness   ASSUMED stakeholder clarification needed   DECIDED submission choice

ID Requirement Source
R1 Accept a request for a specific vehicle, service type, dealership and desired start time. GIVEN
R2 Before confirmation, find one available service bay and one qualified technician for the entire service duration. GIVEN
R3 Persist the confirmed appointment with customer, vehicle, service bay and technician associations. GIVEN
R4 Concurrent requests cannot double-book the vehicle, bay or technician. INFERRED
R5 Service duration is configured by service type; the server computes the end time. ASSUMED
R6 Confirmation is synchronous; replay-safe duplicate requests are outside this implementation milestone. DECIDED

Explicit assumptions

  • One appointment requires exactly one bay and one qualified technician.
  • Appointments use half-open UTC intervals [start, end); back-to-back work is allowed.
  • Service duration is elapsed time: endTime = startTime + duration after converting the requested instant to UTC. Future business-hours rules will be evaluated in the dealership's IANA timezone and local wall-clock, including DST transitions.
  • Bays are interchangeable in the first release; technician qualification depends on service type.
  • Reference data—dealership, vehicles, service types, bays, technicians and skills—is seeded.

Deliberately out of scope

Production authentication, notifications, cancellation/rescheduling and multi-region writes. These do not improve proof of the core scheduling invariant.

3. Architecture and component responsibilities

Component Responsibility Boundary
HTTP adapter Transport validation, correlation IDs, HTTP mapping and OpenAPI. Does not select resources.
Application Validates the command time, orchestrates the use case, emits booking outcome events and invokes the booking port. Does not depend on Prisma types.
Domain Defines interval construction and overlap semantics in pure TypeScript. No framework or I/O imports.
Persistence adapter Owns the PostgreSQL transaction, qualification/allocation queries, locking and constraint-error translation. PostgreSQL-specific behavior stays here.
PostgreSQL Authoritative appointment state and final enforcement of temporal non-overlap. Confirmation exists only after commit.

Technology choices

NestJS
Strong module/DI boundaries and first-class validation/OpenAPI support; appropriate for a small maintainable service.
PostgreSQL
ACID transactions, row locking and GiST exclusion constraints directly support the hardest correctness requirement.
Prisma
Type-safe routine access and migrations. It is deliberately bypassed with parameterized raw SQL only for lock-sensitive PostgreSQL features.
Testcontainers
Concurrency and exclusion semantics must be tested against real PostgreSQL; an in-memory substitute cannot prove them.

4. Booking flow and concurrency correctness

Transaction policy. The booking path uses PostgreSQL READ COMMITTED. It share-locks reference rows for consistency, then exclusively locks compatible bays and qualified technicians in stable ID order. Those resource rows are coordination points; a lock does not represent an abstract time interval.
  1. Begin one adapter-owned transaction; lock and validate reference rows, load service duration and derive the UTC interval.
  2. Lock compatible bays, qualified technicians and qualification rows in stable ID order.
  3. After any wait, re-check database time plus vehicle, bay and technician overlaps for the complete interval.
  4. Select the first available pair, persist and commit. Return 201 only after commit; otherwise roll back and return the mapped 400, 409 or 503 outcome.

Correctness controls

A separate availability read followed by an insert has a TOCTOU race: two requests can observe the same free resources. The authoritative decision therefore stays inside one transaction.

Control Purpose Guarantee
DTO and time validation Reject malformed or invalid commands. Input correctness, not concurrency safety.
Share-locked references Keep referenced definitions stable during allocation. Reference consistency.
Candidate locks + post-lock recheck Coordinate competing allocators before selection. Prevents stale availability decisions.
GiST exclusion constraints Reject overlapping vehicle, bay or technician use. Final invariant under every write path.

Locks provide deliberate allocation: a waiter can choose another complete pair instead of using a constraint violation as normal control flow. Exclusion constraints remain the final authority for application bugs and bypass writes. Stable lock order reduces the known cycle but does not prove the whole system deadlock-free. Proven transient contention maps to 503; exclusion violation 23P01 maps to 409 and is not retried.

Allocation policy

Filter compatible bays and qualified technicians, remove resources with overlapping confirmed appointments, then choose the first free bay and technician in stable ID order. This is deterministic and testable; least-loaded or fair scheduling is a future policy behind the same allocation port.

Rejected alternatives

Alternative Why not for this slice
SERIALIZABLE Correct, but makes serialization abort and retry part of the normal write model; explicit resource locking is easier to demonstrate here.
SKIP LOCKED Can skip resources held by a transaction that later rolls back and produce timing-dependent false conflicts.
Redis/distributed lock Adds a second coordination system while PostgreSQL already owns authoritative state.
Microservices / CQRS / Kafka No independent ownership or asynchronous workflow justifies splitting the consistency boundary.

5. Logical data model and database invariants

The ERD shows the minimum relational model needed to prove the booking contract. It intentionally omits audit columns and secondary indexes so the reviewer can see ownership, qualification and resource assignment at a glance.

Database invariants

  • end_time > start_time.
  • Confirmed intervals cannot overlap for the same vehicle_id.
  • Confirmed intervals cannot overlap for the same service_bay_id.
  • Confirmed intervals cannot overlap for the same technician_id.
  • The assigned technician must be qualified and all assigned resources belong to the requested dealership; these cross-table rules are validated inside the locked transaction.

status is retained because future cancellation must release capacity without deleting appointment history; temporal exclusion applies only to capacity-holding states such as CONFIRMED.

6. API contract and failure semantics

The implementation exposes the smallest reviewable slice that proves durable booking, read-back and operational readiness.

Availability requirement. R2 is satisfied by the authoritative availability check inside POST /appointments. The brief does not require a separate read endpoint; a standalone advisory GET /availability is deferred as a UX optimization.

The assessment API accepts customerId in the body and verifies vehicle ownership. A production API should derive customer identity from an authenticated principal. startTime must be a future instant.

Errors

All failures use application/problem+json (RFC 9457) with a stable problem type and correlation ID.

Status Meaning Example
400 The command is malformed or semantically invalid. Invalid timestamp, unknown field, non-UUID id, or a startTime in the past.
404 A required reference cannot be resolved in the request scope. Unknown vehicle or service type.
409 · RESOURCES_UNAVAILABLE The vehicle is already booked for the interval, or no complete bay and qualified-technician pair remains. Client selects another start time.
409 · REFERENCE_CONFLICT The referenced entities are inconsistent, such as a vehicle that belongs to another customer. Client corrects the request references.
503 · TRANSIENT_FAILURE A proven PostgreSQL deadlock, lock timeout or statement cancellation, or Prisma P2034; the command itself is valid. Generic P2028 errors are not classified wholesale as retryable. Client may retry after the Retry-After hint. Internal bounded retry needs an explicit budget, backoff and observability, so it is deferred.
500 Unexpected failure; internals are not exposed. Correlation ID returned for support.

Deferred: idempotent creation

A later milestone may persist an Idempotency-Key and canonical request hash in the booking transaction, replaying the original response for an identical retry and rejecting key reuse with a different payload.

7. Quality attributes, operations and verification

Verification strategy

Test What it proves Environment
Interval boundaries Containment and partial overlap conflict; touching intervals do not. Pure unit tests
Qualification/allocation Only a valid bay and qualified technician can be selected. Application + PostgreSQL
Database constraint Direct vehicle, bay and technician conflicts are rejected even when the application is bypassed. Real PostgreSQL
Concurrent race Two transactions compete for one pair: exactly one commits, the other returns a conflict, one confirmed row remains. Testcontainers (gateway barrier)
Second-pair allocation A different vehicle with an overlapping booking falls through to the next free bay and technician. The same vehicle is rejected independently of spare resources. Testcontainers
Blocked concurrent mutation While a booking holds its locks, a competing vehicle reassignment, service-type change or qualification revocation must wait for commit (proven via lock timeout). Real PostgreSQL
Driver error contract The real adapter error shapes for exclusion violations (23P01) and lock timeouts are recognized by the failure classification, not assumed. Real PostgreSQL

Observability

Structured JSON logging with request correlation is implemented. The metrics and tracing below are the target design and are deferred to the next milestone.

  • Logs (implemented): structured JSON with a per-request correlation ID echoed in the x-request-id response header, plus a structured booking outcome event (booking_confirmed / booking_rejected with the rejection code, dealership and service-type IDs, and duration). HTTP request logs redact Authorization and Cookie; unexpected errors omit raw messages and stacks.
  • Metrics (planned): booking_creation_total{result}, booking_creation_duration_seconds, booking_conflict_total{resource_type}, booking_transaction_retry_total{reason}, connection-pool saturation.
  • Tracing (planned): HTTP → validation → allocation → insert → commit, with lock/retry events.
  • Initial alerts (planned): error-budget burn, contention rate and pool saturation, with thresholds calibrated only after a load-test baseline exists.

Capacity assumptions and evolution thresholds

The initial sizing hypothesis is 5–20 bays, 10–50 qualified technicians and a few booking writes per second per dealership. A 500 ms P95 booking target is unmeasured and must be validated by load tests. Sustained contention or a missed latency target is the trigger to replace per-candidate overlap checks with a set-based query and narrow lock scope behind the existing allocation port; correctness still ends at PostgreSQL exclusion constraints.

8. AI collaboration, ownership and scope control

I use AI through a Frame → Challenge → Verify → Own workflow: frame the invariant, challenge suggestions for hidden assumptions and failure modes, verify accepted claims against the brief, primary sources and executable artifacts, then own the final decision and wording.

AI output My challenge Decision Evidence
Check availability, then insert the appointment. Two requests can observe the same free pair before either insert commits: a TOCTOU race. Lock candidates, re-check after any wait, then retain exclusion constraints as the final guard. PostgreSQL concurrency tests and exclusion migrations.
Classify every Prisma P2028 as retryable. A generic transaction error does not prove a transient PostgreSQL failure. Retry semantics are limited to known SQLSTATEs and Prisma P2034. Classifier unit tests and a real lock-timeout integration test.
Allow the same vehicle when a different bay and technician are available. This violates the document's central invariant. Add a vehicle-time exclusion constraint independent of resource availability. Migration 202607130003 and PostgreSQL integration tests.
Data boundary. No secrets, production credentials or customer data were supplied to AI tools. AI output is treated as an untrusted proposal, not evidence.
